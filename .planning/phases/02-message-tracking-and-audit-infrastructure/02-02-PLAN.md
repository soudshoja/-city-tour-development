**Step A — Create the middleware class.**

Create `app/GraphQL/Middleware/ResayilContextMiddleware.php`:

```php
namespace App\GraphQL\Middleware;

use Closure;
use Nuwave\Lighthouse\Support\Contracts\GraphQLContext;
```

This is a Lighthouse GraphQL middleware (NOT a Laravel HTTP middleware). It implements `\Nuwave\Lighthouse\Support\Contracts\Middleware` (Lighthouse-specific interface), which requires a `handle(Request $request, Closure $next): Response` signature — check the installed Lighthouse version's middleware interface. For Lighthouse 6.x, the correct approach is to register via `lighthouse.php` config `middleware` array as a standard Laravel HTTP middleware; Lighthouse passes the full HTTP request through the same middleware stack.

Use the simpler approach: create a standard Laravel HTTP middleware (implements `\Illuminate\Contracts\Http\Kernel` style) that runs on the `/graphql` route:

```php
namespace App\GraphQL\Middleware;

use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

class ResayilContextMiddleware
{
    public function handle(Request $request, Closure $next): Response
    {
        // Extract headers (case-insensitive via Laravel's header() method)
        $messageId = $request->header('X-Resayil-Message-ID');
        $quoteId   = $request->header('X-Resayil-Quote-ID');

        // Attach to request attributes for resolver access
        $request->attributes->set('resayil_message_id', $messageId);
        $request->attributes->set('resayil_quote_id', $quoteId);

        return $next($request);
    }
}
```

**Step B — Register the middleware in Lighthouse config.**

Check if `config/lighthouse.php` exists (run `ls config/lighthouse.php`). If it does, add `\App\GraphQL\Middleware\ResayilContextMiddleware::class` to the `middleware` array in that config file. If the config file does not exist, publish it first:

```bash
php artisan vendor:publish --tag=lighthouse-config
```

Then add to the `middleware` array.

If `config/lighthouse.php` does not have a `middleware` key, add:

```php
'middleware' => [
    \App\GraphQL\Middleware\ResayilContextMiddleware::class,
],
```

**Step C — Create or update LighthouseServiceProvider if needed.**

Check if `app/Providers/LighthouseServiceProvider.php` exists. If it does not exist, do NOT create it — the middleware registration via `config/lighthouse.php` is sufficient. Only create `LighthouseServiceProvider.php` if the lighthouse config approach does not work (middleware array not present in installed version).

**How resolvers access the headers:** Lighthouse resolvers receive a `$context` argument that is a `\Nuwave\Lighthouse\Support\Contracts\GraphQLContext` instance. The context object wraps the Request. Resolvers can access the headers via `$context->request()->attributes->get('resayil_message_id')`.

In `DotwService` (Task 2), we will not use context directly — we will accept `resayil_message_id` and `resayil_quote_id` as optional parameters on the four operation methods. The GraphQL resolver is responsible for extracting these from context and passing them down. Since DotwService is currently called directly from `SearchDotwHotels.php`, we are adding parameters to the service methods, not reaching into context from the service (separation of concerns).

  <name>Task 2: Instrument DotwService — audit log calls for all four operation types</name>
  <files>app/Services/DotwService.php</files>
  <action>
Update `app/Services/DotwService.php` to call `DotwAuditService::log()` after every operation. This is an additive change — do not modify existing XML building or response parsing logic.

**Step A — Inject DotwAuditService.**

Add a private property `private DotwAuditService $auditService;` and inject it in the constructor:

```php
use App\Services\DotwAuditService;

public function __construct(?DotwAuditService $auditService = null)
{
    // existing __construct() code unchanged
    // ...

    $this->auditService = $auditService ?? new DotwAuditService();
}
```

This keeps backward compatibility (no breaking change for callers that do `new DotwService()`).

**Step B — Add optional context parameters to the four public operation methods.**

Add two optional parameters at the END of each of these four method signatures (after existing params, with defaults of null):

- `searchHotels(array $params, ?string $resayilMessageId = null, ?string $resayilQuoteId = null, ?int $companyId = null): array`
- `getRooms(array $params, bool $blocking = false, ?string $resayilMessageId = null, ?string $resayilQuoteId = null, ?int $companyId = null): array`
- `confirmBooking(array $params, ?string $resayilMessageId = null, ?string $resayilQuoteId = null, ?int $companyId = null): array`
- `saveBooking(array $params, ?string $resayilMessageId = null, ?string $resayilQuoteId = null, ?int $companyId = null): array`

**Step C — Add audit log calls after each successful operation.**

In `searchHotels()`, after `$hotels = $this->parseHotels($response)` and before `return $hotels`, add:

```php
$this->auditService->log(
    DotwAuditService::OP_SEARCH,
    $params,
    $hotels,
    $resayilMessageId,
    $resayilQuoteId,
    $companyId
);
```

In `getRooms()`, after `$rooms = $this->parseRooms($response)` and before `return $rooms`, add:

```php
$operationType = $blocking
    ? DotwAuditService::OP_BLOCK
    : DotwAuditService::OP_RATES;

$this->auditService->log(
    $operationType,
    $params,
    $rooms,
    $resayilMessageId,
    $resayilQuoteId,
    $companyId
);
```

In `confirmBooking()`, after `$confirmation = $this->parseConfirmation($response)` and before `return $confirmation`, add:

```php
$this->auditService->log(
    DotwAuditService::OP_BOOK,
    $params,
    $confirmation,
    $resayilMessageId,
    $resayilQuoteId,
    $companyId
);
```

In `saveBooking()` (non-refundable path), after `$itinerary = $this->parseItinerary($response)` and before `return $itinerary`, add:

```php
$this->auditService->log(
    DotwAuditService::OP_BOOK,
    $params,
    $itinerary,
    $resayilMessageId,
    $resayilQuoteId,
    $companyId
);
```

Also add to `bookItinerary()` (non-refundable confirmation step), after its parse call and before return, the same `OP_BOOK` log call — this method is the actual confirmation for non-refundable bookings and must also produce an audit row.

**Important:** The `$params` array passed to `searchHotels()` does NOT contain credentials — credentials are inside the XML wrapper built by `wrapRequest()` and are never in `$params`. The `DotwAuditService` sanitizer strips password-like keys as a defense-in-depth measure, but the primary credential isolation is that `$params` never contains them. This satisfies MSG-05.

Do NOT modify the error branches — audit logging only occurs on successful operation completion (after parse). Failed operations are already logged by the existing `$this->logger->error()` calls.

Update PHPDoc on each modified method to document the three new parameters.
