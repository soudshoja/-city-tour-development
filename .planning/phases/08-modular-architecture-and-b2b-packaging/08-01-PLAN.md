**MOD-02 — config/dotw.php is environment-agnostic:**

Run:
```bash
# Check for hardcoded values (non-env() calls) in config/dotw.php
grep -n "=>\s*['\"]" config/dotw.php | grep -v "env("
```

Expected: Any non-`env()` values should be array keys, comments, or static constants (like `rate_basis_codes` array). The runtime-configurable values (`username`, `password`, `company_code`, `dev_mode`, `timeout`, etc.) must ALL use `env()`.

Specifically verify these keys use `env()`:
- `username`, `password`, `company_code`, `dev_mode`
- `request.timeout`, `request.connect_timeout`
- `allocation_expiry_minutes`, `b2c_markup_percentage`
- `cache.ttl`, `cache.prefix`
- `log_channel` — this can be a static string 'dotw' (the channel name is not environment-specific)

If any runtime-configurable key has a hardcoded non-env value, wrap it in `env('DOTW_KEYNAME', 'default')`.

**MOD-03 — Migrations are standalone and idempotent:**

Verify `down()` methods use `Schema::dropIfExists()` (not `Schema::drop()` which throws if table missing):
```bash
grep -A5 "function down" \
  database/migrations/2026_02_21_033317_create_dotw_prebooks_table.php \
  database/migrations/2026_02_21_033318_create_dotw_rooms_table.php \
  database/migrations/2026_02_21_100001_create_company_dotw_credentials_table.php \
  database/migrations/2026_02_21_100001_create_dotw_audit_logs_table.php \
  database/migrations/2026_02_21_165035_create_dotw_bookings_table.php
```

Expected: All use `Schema::dropIfExists()`. If any use `Schema::drop()`, replace with `Schema::dropIfExists()` — this is a one-line fix per file.

For the addColumn migration (`2026_02_21_155718_add_company_id_resayil_to_dotw_prebooks_table.php`), verify the `down()` uses `Schema::table()` with `dropColumn()` only if the columns exist (or uses `dropColumnIfExists()` which is not a standard Laravel helper — check if it uses `hasColumn` guard or simply `dropColumn`). If it uses bare `dropColumn` without guard, wrap with:
```php
if (Schema::hasColumn('dotw_prebooks', 'company_id')) {
    $table->dropColumn(['company_id', 'resayil_message_id']);
}
```

**MOD-04 — graphql/dotw.graphql is independently importable:**

Verify:
```bash
# schema.graphql must import dotw.graphql
grep "^#import dotw.graphql" graphql/schema.graphql

# dotw.graphql must use 'extend type Query/Mutation' (not 'type Query/Mutation')
# so it doesn't conflict with main schema's Query/Mutation types
grep "^type Query\|^type Mutation\|^extend type Query\|^extend type Mutation" graphql/dotw.graphql
```

Expected: `graphql/schema.graphql` has `#import dotw.graphql`. `graphql/dotw.graphql` uses `extend type Query` and `extend type Mutation` (not bare `type Query` which would clash with main schema).

If `dotw.graphql` uses bare `type Query` or `type Mutation`, replace with `extend type Query` / `extend type Mutation`. The schema file in this project currently uses `extend type Query` — confirm this is correct.
