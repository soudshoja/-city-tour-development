<objective>
Build the payment pipeline: PaymentBridgeService generates MyFatoorah payment links, PaymentCallbackController handles webhooks, ConfirmBookingAfterPaymentJob queues the re-block + DOTW confirm + task/invoice creation flow. Adds the payment_link endpoint to BookingController and the payment callback route.

Purpose: Enables B2B-no-credit and B2C booking flows where payment must precede DOTW confirmation. The re-block-after-payment pattern is the most critical architectural piece -- it bridges the gap between DOTW's 3-minute allocation window and payment gateway processing time.
Output: PaymentBridgeService, PaymentCallbackController, ConfirmBookingAfterPaymentJob, payment_link endpoint, callback route.
</objective>

    **1. Create ConfirmBookingAfterPaymentJob** at `app/Modules/DotwAI/Jobs/ConfirmBookingAfterPaymentJob.php`:
    Namespace: `App\Modules\DotwAI\Jobs`

    Implements `ShouldQueue`. Uses `Dispatchable, InteractsWithQueue, Queueable, SerializesModels`.

    Properties:
    - `public int $tries = 4;` -- 4 attempts total
    - `public array $backoff = [30, 120, 300];` -- Exponential backoff: 30s, 2min, 5min between retries
    - `private string $prebookKey;`

    Constructor: accepts `string $prebookKey`.

    **Method `handle(): void`:**
    1. Load booking: `$booking = DotwAIBooking::where('prebook_key', $this->prebookKey)->first()`. If not found, log and return.

    2. Idempotency gate (Pitfall 3 from RESEARCH): if `$booking->confirmation_no` is set, log "already confirmed" and return. Do NOT proceed.

    3. Update status to 'confirming': `$booking->update(['status' => 'confirming'])`.

    4. Call `$bookingService->confirmAfterPayment($booking)` where $bookingService is resolved from container: `$bookingService = app(BookingService::class)`.

    5. If result has error with `needs_refund = true`:
       - If code is 'RATE_UNAVAILABLE': Rate is gone. Initiate refund.
         - Get clientId from CreditService, call `$creditService->refundCredit()` if credit was used (unlikely for this path but handle).
         - Update booking: status='failed', payment_status='refund_pending'.
         - Send WhatsApp notification to customer/agent: "Rate no longer available. Refund initiated." via `WhatsappController::sendToResayil($booking->client_phone ?? $booking->agent_phone, $message)`. Use `app(\App\Http\Controllers\WhatsappController::class)` to get instance.
         - Format message with `MessageBuilderService::formatBookingFailed($booking, 'rate_unavailable')`.
         - Log critical: '[DotwAI] Re-block failed after payment, refund initiated'.
       - If code is 'BOOKING_FAILED': DOTW rejected the booking after re-block succeeded.
         - Same refund flow. Different WhatsApp message: "Hotel could not confirm booking. Refund initiated."
         - Log critical.

    **2. Create PaymentCallbackController** at `app/Modules/DotwAI/Http/Controllers/PaymentCallbackController.php`:
    Namespace: `App\Modules\DotwAI\Http\Controllers`
    Extends `Illuminate\Routing\Controller`

    This controller handles the MyFatoorah payment callback webhook. It does NOT use the dotwai.resolve middleware (payment callbacks come from the gateway, not from WhatsApp users).

    **Method `handleCallback(Request $request): JsonResponse`:**

    1. Extract the payment identifier from MyFatoorah callback. MyFatoorah redirects with `paymentId` query parameter.
       ```php
       $paymentId = $request->query('paymentId');
       ```

    2. Verify payment status with MyFatoorah:
       ```php
       $gateway = new \App\Support\PaymentGateway\MyFatoorah();
       $statusResult = $gateway->getPaymentStatus('payment', $paymentId);
       ```

    3. Extract UserDefinedField to find the DotwAI booking:
       ```php
       $userData = json_decode($statusResult['Data']['UserDefinedField'] ?? '{}', true);
       $prebookKey = $userData['prebook_key'] ?? null;
       ```
       If no prebook_key or process !== 'dotwai_hotel', this isn't a DotwAI payment -- return 200 (ignore gracefully).

    4. Find booking: `$booking = DotwAIBooking::where('prebook_key', $prebookKey)->first()`.
       If not found, log warning and return 200.

    5. Idempotency: If booking status is already 'confirmed' or 'confirming', return 200 (already processing).

    6. Check payment status from gateway response:
       - If payment succeeded (check `$statusResult['Data']['InvoiceStatus']` === 'Paid' or `$statusResult['Data']['InvoiceTransactions']` has a successful transaction):
         - Update booking: `payment_status => 'paid'`, `payment_gateway_ref => $paymentId`.
         - Update the Payment record if it exists: status => 'paid'.
         - Dispatch the queued job: `ConfirmBookingAfterPaymentJob::dispatch($prebookKey)`.
         - Return 200 with `['status' => 'processing']`.
       - If payment failed:
         - Update booking: `payment_status => 'failed'`.
         - Send WhatsApp notification: "Payment was not successful. Please try again."
         - Return 200 with `['status' => 'payment_failed']`.

    7. If `$request->query('error')` is present, handle error flow:
       - Look up booking from UserDefinedField or session.
       - Update payment_status to 'failed'.
       - Return redirect or JSON depending on context.

    Wrap everything in try/catch -- NEVER return non-200 to the payment gateway (they'll keep retrying).
