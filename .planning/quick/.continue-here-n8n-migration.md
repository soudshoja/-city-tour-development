<decisions_made>

- n8n is on separate VPS, Laravel on dedicated server. IP whitelist + Bearer + HTTPS.
- Multi-tenant: Token is system-level. Tenant context (company_id, supplier_id, agent_id) travels in payload.
- Per company+supplier toggle (auto_process_pdf) on supplier_companies pivot — not per agent.
- Reuse existing TaskWebhook pipeline — no new task creation logic needed.
- AIR processor flow stays untouched — only PDF flow changes.
- n8n is tenant-blind — just extracts PDF, returns data. Laravel handles all tenant logic.
- FlyDubai status mapping: 'confirmed' → 'issued' (already in TaskWebhook).
- No invoice created for FlyDubai (only Magic Holiday/TBO trigger invoices).
- IATA wallet auto-deduction for supplier_id=2 + specific agent/IATA combos (already in TaskWebhook).

<context>
The goal is to allow FlyDubai PDF e-tickets to be automatically processed through n8n instead of manual AI extraction. The existing TaskWebhook already handles the complete post-extraction pipeline (task creation, flight details, surcharges, financials, IATA wallet). We just need to connect n8n's callback output to feed into that existing pipeline.

Key insight: FlyDubai has TWO file types — AIR files (keep existing flow) and PDFs (new n8n flow). This change only affects the PDF path.

The comprehensive research doc is at: .planning/quick/flydubai-n8n-pdf-processing-migration.md
</context>
