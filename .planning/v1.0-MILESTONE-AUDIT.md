## Critical Gaps (Blockers)

### 🔴 GAP-1: AUTH-03 — Lighthouse Sanctum Guard Not Configured

**Severity:** P1 — Blocker
**File:** `config/lighthouse.php` line ~67
**Current:** `'guards' => null` → falls back to `web` (session) guard
**Required:** `'guards' => ['sanctum']`

**Impact:** Every N8N `Authorization: Bearer <token>` call to `/graphql` returns `null` from `auth()->user()`. All 4 resolvers detect `$companyId === null` and return `CREDENTIALS_NOT_CONFIGURED`. The entire B2B n8n integration is non-functional.

**Fix:**
```php
// config/lighthouse.php
'guard' => ['sanctum'],
// or in the guards array
'guards' => ['sanctum'],
```

---

### 🔴 GAP-2: BLOCK-02 / GQLR-02 — Invalid DotwErrorAction Enum Value

**Severity:** P2 — Blocker
**File:** `app/GraphQL/Mutations/DotwBlockRates.php` line 87
**Current:** `errorResponse('VALIDATION_ERROR', 'allocationDetails is required', 'RESUBMIT')`
**Required:** Replace `'RESUBMIT'` with `'RETRY'` (semantically correct)

**Impact:** When `blockRates` is called without `allocation_details` (the BLOCK-02 validation path), Lighthouse attempts to coerce `"RESUBMIT"` to `DotwErrorAction` enum and fails with a schema validation error. The caller receives an internal Lighthouse error instead of `VALIDATION_ERROR`.

**Fix:**
```php
// DotwBlockRates.php line 87
return $this->errorResponse('VALIDATION_ERROR', 'allocationDetails is required', 'RETRY');
```
