**CRITICAL OBSERVATION**: This code increments the sequence **AFTER** the invoice is created. However, the invoice was already created in Step 3 using the `invoiceNumber` from the request (which was generated by the `create()` page load). The `$invoiceNumber` generated here on line 1281 is stored in a local variable but **never used** -- it is immediately overwritten by the return statement.

This means:
- The sequence is incremented so the NEXT invoice gets a new number
- The current invoice used the number that was pre-generated when the create page loaded
- There is a **race condition**: if two users load the create page at the same time for the same company, they could get the same invoice number, and the second `store()` call would fail on the `UNIQUE` constraint on `invoice_number`

## Invoice Number Generation

**Method**: `generateInvoiceNumber()` (Line 1977-1981)

```php
public function generateInvoiceNumber($sequence)
{
    $year = now()->year;
    return sprintf('INV-%s-%05d', $year, $sequence);
}
```

**Format**: `INV-YYYY-XXXXX` (e.g., `INV-2026-00001`)

**Uniqueness Guarantee**:
- Database level: `invoice_number` column has a `UNIQUE` constraint (migration line 16)
- Application level: Uses `InvoiceSequence` model (per-company auto-incrementing counter)
- **Gap**: No `DB::transaction()` or row-level locking (`lockForUpdate()`) on the sequence read/increment -- theoretically vulnerable to race conditions under high concurrency

**Alternative method** for programmatic invoice creation:
```php
// Line 4840-4849 - Used by autoGenerateInvoice
private function getInvoiceNumberGenerated($companyId): string
{
    $invoiceSequence = InvoiceSequence::firstOrCreate(
        ['company_id' => $companyId],
        ['current_sequence' => 1]
    );
    $currentSequence = $invoiceSequence->current_sequence;
    $invoiceNumber = $this->generateInvoiceNumber($currentSequence);
    $invoiceSequence->current_sequence++;
    $invoiceSequence->save();
    return $invoiceNumber;
}
```

This is a cleaner helper that generates + increments atomically (but still no DB-level locking).

## Transaction Wrapping

### Is there a `DB::transaction()` in `store()`?

**NO.** The `store()` method has NO transaction wrapping.

**Code evidence** (lines 1222-1277): Each operation uses individual `try/catch` blocks:
- Invoice creation: wrapped in its own `try/catch`
- Each InvoiceDetail creation: wrapped in its own `try/catch`
- On InvoiceDetail failure: `$invoice->delete()` is called (soft-delete), but earlier InvoiceDetails are not explicitly cleaned up

**What happens if it fails mid-way?**
1. If Invoice creation fails -> returns error, no cleanup needed
2. If 1st InvoiceDetail succeeds but 2nd fails -> Invoice is soft-deleted, but 1st InvoiceDetail remains in DB (linked to a soft-deleted invoice)
3. If InvoiceSequence increment fails -> Invoice and InvoiceDetails are created but sequence is not incremented (next invoice would get same number)

### Other methods that DO use `DB::transaction()`:
- `savePartial()` (line 849) -- wraps the entire partial creation + COA entry flow
- `autoGenerateInvoice()` (line 4861) -- wraps the full auto-invoice creation including COA
- Various update methods

## Task Association

### How Are Tasks Selected?

1. User navigates to task list, selects tasks, clicks "Create Invoice"
2. Task IDs are passed as query params: `GET /invoices/create?task_ids=1,2,3`
3. `create()` method validates each task:
   - Must exist
   - Must NOT already have an `invoiceDetail` (not already invoiced)
   - Must be `is_complete = true`
   - Must not be a blocked supplier + status combination
4. Tasks are displayed in the create form with editable `invprice` (invoice price)
5. On form submit, tasks are sent as array to `store()`

### How is InvoiceDetail Created?

For each task in the request array:
1. Fetch the Task from DB to get `supplier_price` (the `total` field on Task)
2. Fetch Supplier, Client, Agent to validate they exist
3. Create InvoiceDetail with:
   - `task_price` = user-entered selling price (`invprice` from request)
   - `supplier_price` = task's total from DB
   - `markup_price` = task_price - supplier_price
   - `profit` = same as markup_price (recalculated later when payment is set)

| `supplier_price` | `task.total` (from DB) |

## Critical Observations

### 1. NO Transaction Wrapping in store()

The `store()` method does NOT use `DB::transaction()`. This means if the process fails mid-way (e.g., second InvoiceDetail fails), the database will be in an inconsistent state: an invoice exists with only some of its details. The manual `$invoice->delete()` on failure is a soft-delete, not a hard rollback.

**Contrast**: The `autoGenerateInvoice()` method (line 4861) and `savePartial()` (line 849) both properly use `DB::transaction()`.

### 2. Invoice Number Race Condition

The invoice number is generated when the `create()` page loads (read from sequence) but the sequence is only incremented when `store()` is called. If two users load the create page concurrently for the same company, both see the same invoice number. The second `store()` call will fail on the `UNIQUE` constraint. There is no `lockForUpdate()` or atomic increment.

### 3. Invoice Number Generated Twice in store()

Lines 1279-1283 generate a NEW invoice number after the invoice is already created (Step 5), but this generated number is never used. The sequence IS incremented, which is the intended side effect, but the code is confusing. The local variable `$invoiceNumber` is overwritten and then discarded.

### 4. COA Entries Are NOT Created During store()

The `store()` method only creates Invoice + InvoiceDetails. No Transaction or JournalEntry records are created. These are created later in `savePartial()` when the user sets up payment on the invoice edit page. This is by design -- the invoice starts as `unpaid` with no payment type.

### 5. No Validation That Tasks Belong to Same Agent/Client

The `store()` method validates that each task's agent/supplier/client exist individually, but does NOT validate that all tasks belong to the same agent or client. The invoice is linked to a single `agent_id` and `client_id`, but individual tasks could theoretically reference different agents/clients.

### 6. Profit vs Markup Price Confusion

During `store()`, both `markup_price` and `profit` are set to the same value (`invprice - supplier_total`). The actual profit (which accounts for gateway charges and agent deductions) is only calculated later in `addJournalEntry()` when payment is processed. The initial `profit` value in InvoiceDetail is misleading.

### 7. Error Response Inconsistency

The `store()` method returns different response formats on error:
- Line 1216-1219: `{ success: false, message: '...' }` (HTTP 200)
- Line 1254: `'Something went wrong'` as plain string (HTTP 404)
- Line 1274: `'Something Went Wrong'` as plain string (HTTP 500)
- Line 1285-1289: `{ success: true, message: '...', invoiceId: X }` (HTTP 200)

### 8. SoftDeletes on All Models

Invoice, InvoiceDetail, JournalEntry, and Transaction all use `SoftDeletes`. The `$invoice->delete()` on line 1272 performs a soft delete, meaning the record remains in the database with a `deleted_at` timestamp. Any orphaned InvoiceDetails pointing to this soft-deleted invoice are still queryable.

### 9. payment_id is Validated but Never Used

The `store()` validation includes `'payment_id' => 'nullable|integer'` but this field is never used anywhere in the method body and is not part of the Invoice::create() call.

### 10. Booking Revenue Account Auto-Creation

The `addJournalEntry()` method (line 1362-1396) will auto-create new COA accounts (e.g., "Cruise Booking Revenue") if they do not exist for a company. This happens silently during payment processing and could create unexpected accounts in the chart of accounts.
